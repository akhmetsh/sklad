# Контекстный файл для планирования и разработки складской программы

## Назначение файла

Этот файл предназначен для передачи полного исходного контекста разработчику, техническому ассистенту или следующей ИИ-сессии, которая будет составлять подробный план разработки и затем помогать писать программу для системы складского учета. Файл не является окончательным техническим заданием, календарным планом или жесткой архитектурной спецификацией. Его задача — зафиксировать предметную область, минимально необходимый объем, предпочтительный технологический стек, логику модулей, базовые сущности, правила учета и примерную последовательность реализации так, чтобы дальнейшее планирование и программирование выполнялись согласованно с уже разработанным контекстом дипломного проекта.

При последующей работе этот файл следует использовать как исходную основу. Разработчик может уточнять детали реализации, оптимизировать структуру проекта, выбирать конкретные UI-компоненты и способ организации кода, но не должен менять предметный смысл системы: это внутренняя веб-ориентированная информационная система для управления складским учетом товарных позиций.

---

## 1. Общая идентичность проекта

### Рабочая тема

**Проектирование и разработка информационной системы для управления складским учетом товарных позиций.**

### Смысл проекта

Система должна рассматриваться как потенциально применимый программный продукт для реальных складских процессов. Ее нельзя описывать как условную учебную демонстрацию, упрощенную имитацию или систему, созданную только ради формального выполнения дипломной работы. Ограничение объема допустимо, но оно должно объясняться логикой минимально жизнеспособной версии, а не академическим упрощением.

### Тип продукта

Внутренняя веб-система административного типа для учета товарных позиций, движения товаров, контроля остатков, ведения справочников, разграничения доступа, формирования отчетов и сохранения истории операций.

### Что система не должна представлять собой

Система не является интернет-магазином, публичным маркетплейсом, CRM-платформой, полноценной ERP-системой, бухгалтерской системой, мобильным приложением или сайтом-визиткой. Она должна быть сосредоточена на складском контуре: товары, склады, места хранения, поступления, выдачи, перемещения, списания, инвентаризация, остатки, отчеты и действия пользователей.

## 2. Главная цель системы

Главная цель системы — автоматизировать основные операции складского учета и предоставить пользователям надежные, своевременные и проверяемые данные о товарных остатках и движениях.

Система должна помогать решать следующие задачи:

- снизить зависимость от Excel, бумажных журналов и разрозненных файлов;
- уменьшить количество ошибок ручного учета;
- обеспечить централизованное хранение данных о товарах, складах, поставщиках и документах;
- фиксировать каждую значимую складскую операцию как документ;
- формировать историю движений товара;
- показывать актуальные остатки по товарам, складам и местам хранения;
- ограничивать доступ к функциям по ролям пользователей;
- создавать основу для отчетности и управленческого контроля.

Эти элементы могут быть описаны как перспективы развития, но они не должны мешать созданию простой и устойчивой первой версии.

## 4. Предпочтительный технологический стек

Базовый стек уже принят и должен использоваться как основной ориентир:

- **Next.js** — основа веб-приложения, маршрутизации, серверной логики и интеграции клиентской части с backend-слоем.
- **React** — построение интерфейса через компоненты.
- **TypeScript** — типизация сущностей, API-контрактов, форм и бизнес-логики.
- **Tailwind CSS** — быстрое создание чистого административного интерфейса.
- **Auth.js** — аутентификация, сессии и защита закрытых маршрутов.
- **Prisma ORM** — описание схемы данных, миграции и типизированная работа с базой данных.
- **PostgreSQL** — основная реляционная база данных.

### Почему PostgreSQL предпочтительнее NoSQL

Складская система содержит много связанных сущностей: товары, категории, единицы измерения, склады, места хранения, поставщики, пользователи, роли, документы, строки документов, движения товара и журналы действий. Для такой модели важны связи, внешние ключи, транзакции, целостность данных и проверяемость операций. Поэтому реляционная база данных подходит лучше, чем документо-ориентированная модель.

## 5. Архитектурная логика

Система должна строиться как клиент-серверное веб-приложение с тремя логическими слоями.

### 5.1 Presentation layer

Этот слой отвечает за интерфейс в браузере:

- страницы входа;
- dashboard;
- таблицы товаров, складов, поставщиков и документов;
- формы создания и редактирования;
- фильтры и поиск;
- отображение остатков;
- отчеты;
- сообщения об ошибках и успешных действиях.

Интерфейс должен быть не маркетинговым, а административным: простые таблицы, понятные кнопки, аккуратные формы, минимум декоративной сложности.

### 5.2 Application layer

Этот слой отвечает за бизнес-логику:

- проверка сессии пользователя;
- проверка роли и доступа;
- валидация входных данных;
- создание документов;
- проверка доступного остатка;
- создание движений товара;
- пересчет или вывод остатков;
- запись действий в audit log;
- подготовка данных для отчетов.

### 5.3 Data layer

Этот слой отвечает за хранение данных в PostgreSQL:

- справочные данные;
- транзакционные документы;
- строки документов;
- движения товара;
- пользователи и роли;
- журнал действий.

## 6. Главный принцип учета

Систему нужно моделировать вокруг **движений товара**, а не вокруг ручного редактирования поля “остаток”. Остаток должен быть результатом подтвержденных операций. В простейшей реализации допускается хранить текущий остаток в отдельной таблице или вычислять его через `StockMovement`, но любые изменения остатка должны происходить только через контролируемую серверную бизнес-логику.

Правильная логика:

- поступление создает положительное движение;
- выдача создает отрицательное движение;
- перемещение создает два движения: минус на исходном складе и плюс на целевом складе;
- списание создает отрицательное движение с причиной;
- инвентаризация создает корректирующее движение при расхождении фактического и учетного количества.

Неправильная логика:

- пользователь вручную меняет число в поле “остаток” без документа;
- документ создается, но движение товара не записывается;
- товар можно выдать при недостаточном остатке;
- перемещение уменьшает остаток на одном складе, но не увеличивает на другом;
- операции не привязаны к пользователю.

## 7. Пользователи и роли

Для минимальной версии достаточно трех ролей.

### Administrator

Администратор управляет пользователями, ролями, базовыми справочниками и системными настройками. Ему доступны все разделы.

### Storekeeper

Кладовщик выполняет складские операции: поступление, выдача, перемещение, просмотр остатков, участие в инвентаризации. Он не должен иметь полный доступ к управлению пользователями и системными настройками.

### Manager

Менеджер или руководитель просматривает остатки, историю движений и отчеты. В минимальной версии он может иметь доступ только на чтение к большинству справочников и документов.

При необходимости позже можно добавить роли `ProcurementSpecialist`, `Auditor`, `Supervisor`.

## 9. Базовые бизнес-правила

1. Каждая складская операция должна оформляться документом.
2. Каждый подтвержденный документ, влияющий на остатки, должен создавать запись в `StockMovement`.
3. Нельзя выдать товар, если доступный остаток меньше запрошенного количества.
4. Перемещение должно быть атомарным: уменьшение на исходном месте и увеличение на целевом месте должны выполняться в одной транзакции.
5. Критические действия должны быть привязаны к авторизованному пользователю.
6. Исторически значимые документы не следует удалять физически. Лучше использовать статус отмены или деактивации.
7. Справочники должны проверяться перед использованием в документах.
8. Количество товара не может быть отрицательным в форме ввода.
9. Поля товара, склада и места хранения должны ссылаться на существующие записи.
10. Доступ к административным функциям должен зависеть от роли.
11. Изменение справочников должно фиксироваться в журнале действий.
12. Подтверждение документа должно быть отделено от черновика, если это не усложняет минимальную версию слишком сильно.

## 10. Backend-контекст

### 10.1 Общий подход

Backend может быть реализован внутри Next.js через Route Handlers или Server Actions. Для минимальной версии лучше выбирать тот вариант, который проще и понятнее конкретному разработчику. Главное — сохранять серверную валидацию, проверку роли и транзакционность операций.

### 10.2 Основные backend-зоны

- authentication;
- users and roles;
- products;
- categories;
- units;
- warehouses;
- storage locations;
- suppliers;
- receipt documents;
- issue documents;
- transfer documents;
- stock balances;
- reports;
- audit log.

### 10.3 Примерная структура backend-логики

Рекомендуется разделить код на несколько уровней:

- `app/api/...` или server actions для внешних вызовов из интерфейса;
- `lib/auth` для проверки пользователя и роли;
- `lib/db` для Prisma client;
- `lib/validators` для схем валидации;
- `lib/services` для бизнес-логики документов и остатков;
- `lib/audit` для записи действий пользователя;
- `lib/reports` для отчетных запросов.

### 10.4 Валидация

Все формы должны проверяться на клиенте для удобства, но основная защита должна быть на сервере. Сервер должен проверять:

- наличие обязательных полей;
- существование связанных сущностей;
- положительное количество;
- права пользователя;
- достаточный остаток для выдачи и перемещения;
- корректность склада и места хранения;
- статус документа перед подтверждением.

### 10.5 Транзакции

Операции, меняющие складской остаток, должны выполняться через транзакции Prisma. Особенно важно для:

- подтверждения поступления;
- подтверждения выдачи;
- подтверждения перемещения;
- списания;
- инвентаризационной корректировки.

## 12. Логика ключевых пользовательских сценариев

### 12.1 Создание товара

1. Пользователь открывает страницу товаров.
2. Нажимает “Создать товар”.
3. Заполняет название, артикул, категорию, единицу измерения, минимальный остаток.
4. Система проверяет обязательные поля.
5. Система сохраняет карточку товара.
6. Действие записывается в журнал.

### 12.2 Поступление товара

1. Кладовщик открывает раздел поступлений.
2. Создает новый документ поступления.
3. Выбирает поставщика, склад и дату.
4. Добавляет строки товаров с количеством и местом хранения.
5. Сохраняет документ.
6. При подтверждении система создает положительные движения в `StockMovement`.
7. Остатки увеличиваются.
8. Действие записывается в журнал.

### 12.3 Выдача товара

1. Кладовщик открывает раздел выдачи.
2. Создает документ выдачи.
3. Выбирает склад и получателя.
4. Добавляет товары и количество.
5. Система проверяет доступный остаток.
6. Если остатка достаточно, документ подтверждается.
7. Система создает отрицательные движения.
8. Остатки уменьшаются.
9. Действие записывается в журнал.

### 12.4 Перемещение товара

1. Кладовщик создает документ перемещения.
2. Выбирает исходный склад и целевой склад.
3. Указывает товар, исходное место хранения, целевое место хранения и количество.
4. Система проверяет доступный остаток на исходном месте.
5. При подтверждении система в одной транзакции создает `TRANSFER_OUT` и `TRANSFER_IN`.
6. История движения сохраняется.

### 12.5 Просмотр остатков

1. Пользователь открывает страницу остатков.
2. Система показывает товар, склад, место хранения, количество и минимальный порог.
3. Пользователь фильтрует по складу, категории или дефицитным позициям.
4. Руководитель может использовать эту страницу как базовый отчет.

## 15. Тестовые сценарии

### 15.1 Авторизация

- пользователь без входа не может открыть dashboard;
- пользователь с ролью Manager не может открыть управление пользователями;
- Administrator может открыть все разделы.

### 15.2 Справочники

- товар нельзя создать без названия;
- товар нельзя создать без единицы измерения;
- склад нельзя создать без названия;
- место хранения должно быть связано со складом.

### 15.3 Поступление

- поступление с одним товаром увеличивает остаток;
- поступление с несколькими строками создает несколько движений;
- документ сохраняет пользователя, который его создал.

### 15.4 Выдача

- выдача уменьшает остаток;
- выдача запрещается, если товара недостаточно;
- выдача не должна создавать частичные движения при ошибке.

### 15.5 Перемещение

- перемещение уменьшает остаток на исходном месте;
- перемещение увеличивает остаток на целевом месте;
- общий остаток товара после перемещения не меняется;
- операция должна быть атомарной.

### 15.6 Отчеты

- отчет по остаткам показывает актуальное количество;
- фильтр по складу возвращает только данные выбранного склада;
- дефицитные товары определяются через сравнение с `minStock`.

## 23. Финальная базовая формулировка

Разрабатываемая система представляет собой внутреннюю веб-ориентированную информационную систему управления складским учетом товарных позиций. Она реализуется на базе Next.js, React, TypeScript, Tailwind CSS, Auth.js, Prisma и PostgreSQL. Основой системы является учет товарных движений через складские документы, контроль текущих остатков, разграничение доступа по ролям, формирование отчетов и сохранение журнала действий пользователей. Минимальная версия должна быть простой для реализации, но обязана сохранять корректную складскую логику: товарные остатки изменяются только через подтвержденные операции, каждое движение имеет документальное основание, а критические действия привязаны к авторизованному пользователю.
